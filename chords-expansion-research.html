Expanding Chords Mode

Three proposals — scale-tone pedal bends inside a chord, layered “kosher-chord” highlighting, and a keyboard-mapping scheme relocated to Settings — each grounded in what vg800-tuner.html actually does today.

00Executive summary

All three features are feasible and fit the existing architecture well — but each leans on a subsystem that was built for a slightly different purpose, and the friction points are worth naming up front.

  • Area 1 (scale-tone bends) is the most novel and the highest-effort. Your pedal engine (controlState(), bendsFromEngaged(), setCtrl()) already does exactly the right mechanical thing — aggregate per-string semitone offsets, clamp to ±12, send CC, live-preview the destination note. What it lacks is a scale-aware, per-string move generator. The existing copedent computes moves from a detected major triad root (chordInfo()), which is precisely why pedals feel “disabled” under most chords. The fix is a new bend-mode that reads settings.keyRoot/keyScale instead of re-detecting a root, and maps the six pedals to the six strings.
  • Area 2 (layered highlighting) is mostly additive and low-risk. Your diatonicMap() already emits {func, rn} keyed by "<pc>-<rowKey>"; a parallel diatonic7Map() and a small borrowedMap() reuse the same shape. The real design work is visual encoding — using fill for function, border-style for layer, and corner badges for extension/target so four kinds of information coexist without a rainbow. Toggleable layers keep it legible.
  • Area 3 (keyboard + Settings move) is the cleanest. Number keys 1–7 are currently unbound, and diatonicChordAt() already returns exactly the {r, row, formula} that applyChord() needs. Relocating the keymapHTML() drawer from renderPedals() into a Settings group is a UI move, not a logic change — KEYMAP, rebuildKeymap(), and the two global keydown handlers keep working untouched.

▲ Recommended order

Ship in order of risk, not glamour: (1) relocate key-mapping to Settings, (2) add chord number-keys, (3) add the 7ths & borrowed layers, (4) build scale-tone bends last. Rationale in § Implementation order.

01What the code does today (the parts these features touch)

Every proposal below is anchored to these facts, so they’re collected here once.

The chord grid & key mode

  • renderChords() (~L1864) builds a 12-root × N-row matrix. Rows come from chordRows() = [...CHORD_FIXED, oneSpecialRow]. CHORD_FIXED = majmin7min7maj7dimaug; the special row is one of CHORD_EXT (add96m6sus2sus4dim7m7♭5), chosen by the Extend segmented control (settings.chordExt).
  • Diatonic highlighting: when a scale is chosen, diatonicMap(root, scale) returns a dict keyed "<pc>-<rowKey>" → {func, rn}. In renderChords() each cell looks itself up and gets class tonic|subdom|dominant (fill+border) or offkey (dimmed). Roman numeral printed via romanNumeral().
  • Only seven triads light up. SCALES[x].quals holds one quality per degree (the triad), mapped to a grid row via QUAL_ROW. Nothing surfaces diatonic 7ths, secondary dominants, or borrowed chords.
  • Latching one-tap: tapping a cell calls applyChord(r, row, name, formula) → voiceChordVoiced() → apply(), which sets baseOffsets, clears held, and stores activeChord={r,row}. The chord stays until you tap another. There is no momentary chord.
  • ±12 grey-out: a cell is disabled when voiceChord(ri,row.formula) returns null, which happens iff any string’s offset would exceed ±12 (toCC() maps −12…+12 → 0…127).
  • Voicing engine: settings.voicing (Spread/Voiced), voiceSize (3/4), voiceStart (cluster window), voiceFeel (Legato/Grip). applyChord() already honours all of these — anything that calls it inherits the current voicing for free.

The Bends / pedal engine

  • A copedent is a set of controls; each control has moves:[{deg,semis}] expressed in chord degrees ('1'/'3'/'5'). controlState(ctrl, info, base) resolves each degree to a string via the current root, sums per-string bends, and greys the pedal if a target degree is missing or any result exceeds ±12.
  • chordInfo(base, name) finds the root by looking for a major triad (r, r+4, r+7) among the sounding pitch-classes. This is the crux of “pedals are disabled in Chords mode.” A minor/dim/sus voicing contains no major triad at its root, so info.root==null → “No chord root here — pedals disabled.” Even a major chord leaves the pedals doing pedal-steel degree moves, not scale moves.
  • setCtrl() aggregates bendsFromEngaged() onto baseOffsets and calls send(); renderStrings() animates the slide. pressCtrl()/releaseCtrl() implement momentary-vs-latch via settings.latch. Per-pedal preview text comes from movesHTML(st.moves) — “String 3  G→A”.

Keyboard mapping

  • DEFAULT_KEYS = {A:'a',B:'s',LKL:'d',LKR:'f',AB:'g',ABR:'h'}; overrides in settings.keys. rebuildKeymap() builds KEYMAP: char→ctrlId.
  • Two global handlers: a rebind-capture keydown (active only while bindingCtrl is set) and a pedal-engage keydown that ignores repeats, meta/ctrl/alt, open modals, and INPUT/TEXTAREA/SELECT targets, then does KEYMAP[key] → pressCtrl. A matching keyup releases. Number keys 1–7 are not in KEYMAP, so today they do nothing — free to claim.
  • The binding UI (keymapHTML(), the .kdraw “Key Mapping” drawer with the .kb-chip chips and the Latch checkbox) is rendered inside renderPedals(), i.e. it physically lives in the Bends panel.

02Area 1 — Scale-tone bends via the six pedals, inside Chords mode

Re-enable the pedals during Chords mode so each pedal raises one string to the next scale tone above its current pitch while the other strings sustain — a live, partial, pedal-steel voice-leading gesture, distinct from the chord buttons’ full static retune.

2.1 · Why this is musically worth it (and not redundant with sus/add9)

The existing sus2/sus4/add9 rows perform a full six-string retune and latch: every string jumps to the new chord and stays. That is a chord change. A scale-tone bend is a different musical object:

Dimensionsus/add9 row (exists)Scale-tone pedal (proposed)
Strings affectedAll six re-voicedOnly the string(s) whose pedal you hold
DurationLatched — permanent until next tapMomentary — springs back on release
Motion heardA new chord appearsA voice moves under a sustained chord — the “cry”
Note chosenFixed formula (2nd or 4th over root)Whatever the next scale tone is for that string
Gesture familyComping / arrangementOrnament / passing-tone / suspension-and-resolve

Concretely: over a held C major, tapping the pedal for the string sounding E lifts it to F — a 4–3 suspension you resolve by letting go, which the sus4 row can’t do because it also moves the other strings and never returns. This is the core pedal-steel idiom, and it’s the thing your app can’t currently produce.

2.2 · Computing “next scale tone up” per string

The rule is small and deterministic. For string i with current sounding MIDI note m = STD_MIDI[i] + baseOffsets[i], and the active scale’s pitch-class set S (derived from settings.keyRoot + SCALES[settings.keyScale].intervals):

// next scale tone strictly above m, as a semitone delta d in 1..11
function nextScaleUp(m, S){
  for(let d=1; d<=11; d++) if(S.has(((m+d)%12+12)%12)) return d;
  return null;              // unreachable for a non-empty scale
}
// per-string move, clamped by the SAME ±12 rule controlState() already enforces
const off = baseOffsets[i] + nextScaleUp(m, S);   // grey the pedal if off > 12

For a diatonic scale the delta is always 1 or 2 semitones (a half- or whole-step) — exactly the grace-note size that reads as “bend,” never a leap.

Worked example — C Major, holding a C-major triad (Spread voicing)

Using voiceChord(0,[0,4,7]), the six strings sound (string 1 → 6): C4 · G3 · E3 · C3 · G2 · C2. With S = {C,D,E,F,G,A,B}:

Pedal → StringSoundsNext scale tone upΔ semisBecomes
P1 → str 1C4D+2D4
P2 → str 2G3A+2A3
P3 → str 3E3F+1F3
P4 → str 4C3D+2D3
P5 → str 5G2A+2A2
P6 → str 6C2D+2D2

▲ The theory pays off

Press all six and C(C·G·E·C·G·C) becomes D·A·F·D·A·D — a Dm chord (the ii). The whole triad walks up the scale by step. Press only P3 and you get a 4–3 suspension (F over the C triad). Press only P1 and you get an added-9 colour (D over C). These partial results are the point: one pedal set gives you suspensions, passing tones, and stepwise chord motion, all live under the fingers.

2.3 · Is “up to the next scale tone” always the right move?

For a first version, yes — raises only, and keep it uniform. Reasons and caveats:

  • Raises are the idiom. Pedal-steel A/B pedals and most knee levers raise; that’s the sound the ear expects when a note “blooms.” Uniform behaviour is also predictable — the same physical gesture always means “this string, up one scale step.”
  • Leading-tone bonus. A string sounding the 7th degree bends up a half-step to the tonic (B→C) — a built-in resolution. A string on the 3rd bends to the 4th (E→F, the classic suspension). These fall out of the rule for free.
  • Where up is debatable: a string already on the tonic bends to the 2nd (a fine add-9 colour), and a string on the 5th bends to the 6th (also musical). There’s no “wrong” degree, but two refinements are worth considering for v2:
    1. Optional lowers. A companion “lower to next scale tone down” set (a modifier, or a second pedal bank) unlocks the maj7 drop (C→B over C, mirroring the existing LKR “root −½ (maj7)”) and 4→3 from above.
    2. Chord-tone snapping. An alternate mode that bends to the next chord tone rather than the next scale tone gives fatter, always-consonant motion — but that is closer to what the chord rows already do, so scale-tone is the more distinctive default.

Recommendation: ship raises-to-next-scale-tone as the default and only behaviour; expose lowers later behind a modifier if there’s appetite.

2.4 · How this maps onto your pedal engine

◆ Reuse, don’t rebuild

Your engine is already the right shape. The only new part is a move generator that produces per-string {deg?, semis} from the scale instead of from a detected major-triad root.

Introduce a synthetic copedent — call it COPEDENTS.scaleBend — active when Chords key-mode is on. Its six controls map 1:1 to strings:

// conceptual — six string-pedals; moves are computed live, not fixed
scaleBend: { controls: [
  {id:'S1', name:'String 1', group:'string', key:'a', string:0},
  {id:'S2', name:'String 2', group:'string', key:'s', string:1},
  /* … S3 'd', S4 'f', S5 'g', S6 'h' … */
]}

Then either (a) special-case controlState() when the control has a string field, computing nextScaleUp() against settings.key*; or (b) recompute each control’s moves live in renderPedals() before controlState() runs. Crucially, this path must bypass chordInfo() — the scale, not a re-detected chord root, is the source of truth. Everything downstream is unchanged:

  • controlState()’s ±12 clamp and grey-out already handle the range limit — a pedal near the ceiling simply disables itself, no new logic.
  • bendsFromEngaged() already sums multiple held pedals per string, so holding P1+P3 raises two voices at once.
  • movesHTML() already renders “String n  from→to” — the live per-string preview the request asks for is already there; it just needs the scale-derived from/to. Each pedal button will read e.g. “String 3  E→F.”

2.5 · Momentary vs latch, and pairing with chord latching

Confirmed: this must be momentary/hold. The whole gesture is the release. The mechanics already exist: releaseCtrl() restores baseOffsets when settings.latch is false. Two clarifications:

  • The chord stays; the bend is temporary. activeChord and baseOffsets hold the latched chord; pedals add a transient offset on top and remove it on release. This composition is exactly how setCtrl() already layers onto baseOffsets — so “hold a chord, bend within it, let go, chord intact” works with no state fight.
  • Latch interaction is a genuine ambiguity. settings.latch currently governs pedals globally. If a user turns pedal-latch on, a scale-tone bend would stick — musically odd (you’d strand a suspension). Recommendation: for the scale-bend copedent, force momentary regardless of settings.latch, or give the scale-bend bank its own latch that defaults off. Flag either way — see §Flags.

⚑ Difficulty — this feature needs a scale to exist

“Next scale tone” is undefined when settings.keyScale === 'off'. Decide the fallback explicitly: (recommended) disable the scale-bend pedals and show “Pick a key & scale to enable scale bends,” mirroring the existing “No chord root here — pedals disabled” affordance; or fall back to a chromatic +1 raise; or fall back to next chord tone. The first keeps the feature honest to its name.

⚑ Difficulty — six “pedals” that aren’t foot pedals

The Bends UI groups controls into Foot / Knee / Combos with pedal-plate SVG badges (PEDAL_ICONS). Six per-string raises don’t fit that metaphor. Recommend a new group label (e.g. “Scale Bends · one string each”) and a string-coloured badge using STR_COLORS[i] instead of a foot-pedal icon, so the six buttons read as strings, not pedals. This is presentation only; renderPedals() already switches its section headers by group.

03Area 2 — Layered highlighting of more “kosher” chords

Surface the diatonic 7ths, the borrowed chords, and where dominants resolve — as toggleable layers encoded with border, badge, and glow rather than more fill colours, so multiple facts coexist without turning the grid into a rainbow.

3.1 · What’s harmonically available beyond the seven triads

Diatonic 7th chords (per mode)

Each degree has a natural four-note extension. These map almost entirely onto rows you already render — maj7, min7, 7 (dom) are fixed rows; only the half-diminished needs the m7♭5 special row. That’s the whole “7ths layer”: light the correct 7th-quality cell for each degree, with the same function colour as its triad.

◆ Implementation — a parallel map

Add SCALES[x].quals7 (the seven 7th-qualities) and a diatonic7Map() that is a carbon copy of diatonicMap() but reads quals7 and maps quality→row through a small QUAL_ROW7 (maj7→maj7, m7→min7, dom7→sev, m7♭5→m7b5, °7→dim7). renderChords() then adds a class when the 7ths layer is on and a cell matches. No new grid geometry — the cells already exist on the fixed rows.

Chords pulled from the parallel mode — the everyday ones players actually reach for. In a major key the high-value borrowings are:

Other genuinely useful options

  • Suspensions (sus2/sus4) are scale-agnostic and almost always available; better surfaced as an always-on “these work here” hint than as a diatonic claim.
  • 6ths (I6, ii-flavoured 6) — the 6/m6 rows; diatonic when their added 6th is in the scale. Lower priority; fold into the 7ths/extensions layer if desired.
  • Resolution targets — not a chord type but a relationship: every dominant-function chord “wants” to resolve down a fifth. Worth its own layer (below).

3.3 · Toggling — keep it from becoming a rainbow

Layers are opt-in and independent, driven by a compact chip row placed next to the existing function legend (which already lives in .chord-legend). Persist each toggle in settings (e.g. settings.chordLayers = {sevenths:false, borrowed:false, resolutions:false}).

▣ Triads · always on ⑦ 7ths ⌁ Borrowed → Resolutions
  • Default: triads only (today’s behaviour) — zero regression.
  • 7ths adds badges, not colours — safe to leave on permanently for jazz users.
  • Borrowed is the “noisiest” layer; dashed-violet keeps it clearly a different class of suggestion, and it only promotes ~5 cells, not all off-key cells.
  • Resolutions is interaction-driven (on hover/selection of a dominant) rather than always-painted, so it never adds standing clutter.

◆ Maps cleanly onto renderChords()

Each cell’s class string is already assembled inline (cls in the CHORD_ROOTS.forEach loop). Adding layers = OR-ing in a few more classes computed from diatonic7Map()/borrowedMap() and, for badges, appending a <span class="cc-badge">. The resolution ring is a hover/focus handler that toggles a class on the target cell — the cells are already keyed by data-r/data-row, so the target is trivially found (diatonicChordAt(root,scale,0) for a V→I).

⚑ Difficulty — one special row, but many 7ths

The grid shows only one CHORD_EXT row at a time. The diatonic 7ths of a major scale need maj7, min7, and 7 (all fixed rows — fine) plus m7♭5 for viiø7, which is only the special row. So the “7ths layer” can fully light up only when the half-diminished degree’s row is present. Options: (a) auto-set the special row to m7b5 while the 7ths layer is on; (b) promote m7♭5 and dim7 to always-visible fixed rows (bigger grid); (c) accept that viiø7 shows only when the user selects the m7♭5 row, and badge the triad’s dim cell as “→ ø7 available.” Recommend (a) — least visual cost.

⚑ Difficulty — harmonic minor has no home for two of its 7ths

i(mMaj7) and III+(maj7) aren’t expressible on any current row (no min-major-7 or aug-maj-7 row). For the 7ths layer in Harm. Minor, either add those rows, or gracefully fall back to badging the triad (“7th n/a on grid”). Not blocking — just note the gap so the layer doesn’t look broken there.

04Area 3 — Keyboard mapping for chords, and moving key-mapping to Settings

Bind 1–7 to the seven diatonic chords of the current key — voiced with whatever voicing settings are live — using the same latch as the on-screen cells; add sensible modifiers; and relocate the whole key-mapping UI out of Bends into a dedicated Settings area with per-section “Map ⌨” jump buttons.

4.1 · Number keys → diatonic degree chords

The wiring is short because diatonicChordAt() already returns exactly what applyChord() consumes:

// pressing "3" in C Major → the iii chord (Em), current voicing, latched
function playDegree(deg){                 // deg = 0..6
  if(settings.keyScale==='off') return;   // no scale ⇒ no degrees
  const nc = diatonicChordAt(settings.keyRoot, settings.keyScale, deg);
  if(nc) applyChord(nc.r, nc.row, CHORD_ROOTS[nc.r]+nc.suffix, nc.formula);
}

Because it routes through applyChord(), three requested behaviours are free:

  • “Currently active voicing settings” — applyChord → voiceChordVoiced() reads settings.voicing/voiceSize/voiceStart/voiceFeel live. Number keys inherit Spread/Voiced, 3/4-note, Legato/Grip automatically.
  • Same latch as the buttons — applyChord() sets activeChord and clears held; press 1 then 4 and it swaps I→IV, identical to tapping the cells. No keyup handler needed for chords (unlike pedals).
  • Stays in key — diatonicChordAt() returns the diatonic quality per degree, so 2 is a minor ii in major but a diminished ii° in a scale where that degree is diminished.

⚑ Ambiguity — “active quality/extension” vs staying in key

The brief says number keys should use “whatever quality/extension … currently selected.” But the Extend special row can be non-diatonic (e.g. add9, sus4), and blindly applying it to all seven degrees can throw out-of-scale notes. Two coherent readings:

(Recommended) In-key by default: 1–7 play each degree’s diatonic quality (triad), inheriting the voicing engine settings but not the special-row extension. Colour/extensions come from modifiers (below). This guarantees every number key is musically valid in the chosen scale.

Alternative — literal: apply the current Extend row’s formula transposed to each degree root. More “what you see is what you get,” but produces clams on degrees where the extension isn’t diatonic. If you want this, gate it so only diatonic-safe extensions (6, add9) apply, and fall back to the triad otherwise.

I recommend the in-key default and have designed the modifiers around it.

4.2 · A non-conflicting modifier scheme

The pedal-engage handler already ignores ctrlKey/metaKey/altKey, so those are “safe” from the pedals’ perspective — but Ctrl/Cmd collide with the browser and Alt+digit trips OS menus on some platforms. Shift is the clean primary modifier (the pedal handler lets it through, and it never triggers browser chrome). Use e.code (Digit1…Digit7) so Shift’s symbol remapping (Shift+2 = “@”) doesn’t matter.

KeystrokeActionNotes
1–7Play that degree’s diatonic chord, current voicing, latchedCore mapping
Shift+1–7Same degree as its diatonic 7th (4-note)Uses quals7 from Area 2; momentary voice-size bump, doesn’t change the persistent 3/4 setting
0 or `Clear the active chord / re-center LegatoMaps to existing lastVoiced=null + clear
[ / ]Nudge the voicing cluster window down / up= voiceStart ∓1; only meaningful in Voiced mode
- / =Toggle 3-note / 4-note voicing= voiceSize; persistent, mirrors the on-screen seg

On “sus” as a modifier: a dedicated sus toggle is tempting, but the only truly safe modifier left is Alt (with the OS caveat). Recommendation: expose sus through the existing Extend row rather than a keystroke, or — if a key is wanted — make it a sticky on-screen toggle the number keys respect, avoiding Alt entirely. Flag: don’t overload Alt+digit.

⚑ Requirement — degrees need a scale selected

With keyScale==='off' the seven degrees are undefined. Number keys should no-op and the Chords panel should hint “Select a key & scale to enable 1–7,” consistent with how the rest of key-mode gates on keyOn. Don’t silently map 1–7 to chromatic roots — it would contradict the “diatonic” promise.

⚑ Care — number keys inside text fields

The new chord handler must copy the pedal handler’s guards: bail on open modals (.scrim.open) and on INPUT/TEXTAREA/SELECT targets, or typing a channel number / tuning name will fire chords. This is a one-line reuse of the existing guard.

4.3 · Relocating key-mapping to Settings + per-section “Map ⌨” buttons

Today keymapHTML() renders the .kdraw drawer inside renderPedals(), so it’s buried in the Bends panel and knows only about pedals. Move it to a first-class Settings group and let it describe every section that has keystrokes.

Information architecture

Add a <details class="settgrp" data-grp="keys"> named “Keyboard Mapping” to the Settings sheet (alongside MIDI Output, CC numbers, Bend effect), organised by section:

Keyboard Mapping                         (new settgrp in #settingsModal)
├─ Global
│   └─ Latch — tap on/off instead of holding   [✓]
├─ Bends            ← the current .kb-chip rebind chips, verbatim
│   ├─ Pedal A         [ A ]
│   ├─ Pedal B         [ S ]
│   ├─ LKL / LKR …     [ D ] [ F ]
└─ Chords           ← new
    ├─ Degrees 1–7     [ 1 … 7 ]   (fixed, shown for reference)
    ├─ 7th modifier    [ Shift ]
    └─ Voicing nudge   [ [ ] − = ]

Each row shows the key and what it does; the Bends rows keep the exact rebind-on-click behaviour (bindingCtrl capture) that already works. The number-key chord bindings can start fixed (1–7 are conventional) with rebinding as a later nicety.

The “Map ⌨” buttons

Put a small ghost icon-button in each section header that has keystrokes — Bends and Chords (Panning has none). It should match the app’s existing icon-button language (the round .gear / theme buttons, or the small .add button):

  • Look: a compact pill/round button with a keyboard glyph and “Map” label, brass on hover, same border tokens as .gear.
  • Behaviour: on click → openSettings() (which already calls renderSettings() + open('settingsModal')), then programmatically open the Keyboard Mapping <details>, scroll it into view, and briefly highlight the relevant sub-block (Bends vs Chords). openSettings() and the data-grp convention already exist, so this is a targeted expand-and-scroll, not new plumbing.

◆ Keeping every existing binding working after the move

Nothing about the logic moves — only the DOM location of the chips. KEYMAP, rebuildKeymap(), settings.keys, the rebind-capture keydown, and the engage/keyup handlers are all global and unaware of where the chips render. The one real change: renderPedals() currently re-renders on rebind (bindingCtrl → renderPedals()); after the move, the Settings render function owns re-rendering the chips. Extract keymapHTML()’s chip list into a function the Settings group calls, and repoint the rebind handler’s re-render at it. The Bends panel simply stops emitting the drawer. Low risk, mechanical.

⚑ Note — the Latch toggle is shared

settings.latch lives in that drawer today and governs the pedals. If it moves to a “Global” sub-block, keep in mind Area 1 wants scale-bends to stay momentary regardless — so either scope latch per-bank or document that Global Latch doesn’t apply to scale-bends. (Same flag as Area 1 §2.5.)

05Suggested implementation order

  1. Relocate key-mapping to Settings + add “Map ⌨” buttons (Area 3.3). Pure refactor, no music theory, immediately improves IA and de-clutters Bends. De-risks everything after it by giving keystroke config a real home.
  2. Chord number-keys 1–7 + Shift-for-7th (Area 3.1–3.2). Tiny surface — one guarded handler calling diatonicChordAt()/applyChord(). High daily value, no visual change. Depends on quals7, which you’ll want for Area 2 anyway, so add quals7 here.
  3. Area 2 layers, in sub-steps: (a) 7ths badges via diatonic7Map() — reuses quals7 from step 2; (b) resolution rings (interaction-only, no standing clutter); (c) borrowed-chord dashed layer + the layer-toggle chips. Additive, reversible, default-off.
  4. Scale-tone pedal bends (Area 1). Last because it’s the only feature that adds a new subsystem (the scaleBend copedent + its move generator + a non-foot-pedal presentation) and has the most edge cases (scale-off fallback, latch scoping, ±12 at the ceiling). By this point the key/scale plumbing and the quals7/scale-set helpers are already battle-tested.

Each step is independently shippable and leaves the app in a working state; nothing later is blocked by anything but the small shared helpers noted above.
